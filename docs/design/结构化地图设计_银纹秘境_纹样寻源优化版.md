# 《银纹秘境：彝饰守护者》结构化地图设计文档（纹样寻源优化版）

## 当前实现同步说明（2026-07-01）

当前运行版本的 TMX 地图位于 `assets/maps/`，tileset 位于 `assets/tilesets/`。不要移动或重命名这些资源。

当前演示主线为：

```text
村寨长者对话 -> 集市寻源 -> 河谷影纹净化 -> 节庆广场守护 -> Ending
```

文中更完整的纹样空间、工坊修复、图鉴和多场景扩展内容属于预留 / 后续扩展。

## 一、文档说明

### 1.1 文档目的

本文档用于说明《银纹秘境：彝饰守护者》新版地图结构设计，包括主地图、纹样寻源地图、纹样空间小关卡、地图尺寸、视窗规格、瓦片规格、场景连接、地图图层、对象层字段、碰撞区域、NPC 点位、物品点位、自然线索点、纹样检查点、修复台、展示台、摄像机跟随和地图验收标准。

本版本根据新版游戏策划、SRS 和 FSD 进行调整。旧版地图文档重点是“村寨入口—工坊—山地集市—节庆广场”四个主要场景，以及“纹样拓印点”对象。新版地图设计需要支持“纹样寻源系统”，即玩家通过自然场景寻源、走纹样迷宫、纹样对齐或关键点描摹获得纹样拓片，再返回工坊完成敲银修复。

### 1.2 文档定位

本文档是游戏策划、软件需求规格说明书和功能规格说明书的地图落地文档。

它主要回答以下问题：

1. 新版游戏需要哪些主地图与小关卡地图；
2. 每张地图的尺寸、瓦片、图层和对象层如何设置；
3. NPC、物品、纹样寻源点、自然线索点、检查点、修复台和展示台如何放置；
4. 纹样寻源玩法如何通过 Tiled 地图和对象层实现；
5. 玩家如何从主地图进入纹样空间，又如何返回原地图；
6. 摄像机、碰撞和世界坐标如何保持统一；
7. 地图如何支撑“银饰套装修复、拓片收集、敲银修复、银光值评价”的新版流程；
8. Codex/AI 后续应如何按照地图文档生成代码。

### 1.3 地图设计原则

本项目地图设计遵循以下原则：

1. 地图必须服务游戏流程，而不是只做背景图；
2. 每个地图都要有明确功能，例如教学、寻源、材料收集、修复或结局展示；
3. 主地图尺寸应大于视窗，体现 2D 滚轴探索；
4. 纹样空间地图可以比主地图小，但仍应支持碰撞、检查点、返回点和 UI 提示；
5. 地图尺寸应尽量为 32×32 瓦片尺寸的整数倍；
6. 地图应分为背景层、地面层、装饰层、碰撞层、对象层、角色层和前景层；
7. NPC、物品、纹样寻源点、自然线索点、检查点、场景出口等不应全部写死在代码中，应尽量通过 Tiled 对象层配置；
8. 角色移动、碰撞检测和交互判断应基于世界坐标；
9. 屏幕显示应通过摄像机将世界坐标转换为屏幕坐标；
10. 主地图路线要清晰，不做复杂迷宫；纹样空间可以做“小型纹样迷宫”，但必须短、明确、可重试；
11. 地图难度应适合小学生、初中生和课程展示体验者；
12. 纹样来源表达采用“游戏化转译”，自然意象和纹样关系用于游戏机制表达，不写成未经证实的绝对民俗结论。

## 二、全局地图规格

### 2.1 视窗规格

游戏主视窗建议设置为：

```text
800 × 600 像素
```

设计原因：

1. 适合 PC 端课程展示；
2. Pygame 实现难度适中；
3. 便于 UI/HUD 布局；
4. 能清楚展示角色、NPC、纹样点和地图细节；
5. 不会给素材制作带来过大压力。

### 2.2 瓦片规格

地图瓦片统一建议设置为：

```text
32 × 32 像素
```

设计原因：

1. 32×32 是常见 2D Tile Map 尺寸；
2. 适合像素风或简化手绘风；
3. 方便在 Tiled 中制作地图；
4. 方便对象定位和检查点摆放；
5. 方便碰撞检测；
6. 方便计算地图像素大小。

### 2.3 地图尺寸规范

地图像素大小应尽量是 32 的整数倍。

推荐尺寸：

```text
3840 × 2560    120 × 80 tiles
2560 × 1920     80 × 60 tiles
1920 × 1280     60 × 40 tiles
1280 × 960      40 × 30 tiles
960 × 768       30 × 24 tiles
```

不推荐尺寸：

```text
3780 × 2395
```

原因：

1. 不能被 32 整除；
2. 容易导致瓦片错位；
3. 影响碰撞区域对齐；
4. 影响对象点位计算；
5. 影响摄像机边界计算。

### 2.4 坐标体系

本项目采用两套坐标体系。

#### 世界坐标

世界坐标表示对象在完整地图中的真实位置。

```text
player_world_x = 1600
player_world_y = 1200
```

#### 屏幕坐标

屏幕坐标表示对象显示在当前视窗中的位置。

```text
screen_x = world_x - camera_x
screen_y = world_y - camera_y
```

所有 NPC、道具、纹样寻源点、自然线索点、检查点、触发区和碰撞区域都应以世界坐标存储和判断。

### 3.3 地图连接关系

主地图连接关系：

```text
village
 ├── workshop
 ├── market
 ├── mountain
 └── festival
```

推荐主流程：

```text
village → workshop → mountain / market → pattern_xxx → workshop → festival
```


4. 进入纹样空间小关卡，获得拓片；
5. 返回银饰工坊敲银修复；

## 四、地图图层规范

### 4.1 标准图层结构

每张主地图建议包含以下图层：

```text
background
ground
decoration
collision
object_points
npc_points
pattern_points
clue_points
trigger_points
foreground
```

纹样空间地图建议包含以下图层：

```text
background
ground
pattern_path
collision
checkpoint_points
trigger_points
foreground
```

### 4.2 图层职责说明

| 图层名 | 类型 | 用途 |
|---|---|---|
| `background` | Tile Layer | 远景、底色、天空、水面底图等 |
| `ground` | Tile Layer | 草地、道路、室内地板、山路等可行走区域 |
| `decoration` | Tile Layer | 房屋、摊位、树木、工具架、灯笼等装饰 |
| `pattern_path` | Tile Layer | 纹样空间中的发光路径、山脊线、水波线等 |
| `collision` | Object Layer | 不可通行区域 |
| `object_points` | Object Layer | 道具、修复台、展示台、隐藏银纹等 |
| `npc_points` | Object Layer | NPC 出生点 |
| `pattern_points` | Object Layer | 纹样寻源入口 |
| `clue_points` | Object Layer | 自然场景线索点 |
| `checkpoint_points` | Object Layer | 纹样空间检查点 |
| `trigger_points` | Object Layer | 出生点、出口、剧情触发区、返回点 |
| `foreground` | Tile Layer | 树冠、屋檐等前景遮挡 |
| UI/HUD | 程序绘制 | 当前任务、套装进度、银光值、图鉴进度 |

### 4.3 图层绘制顺序

推荐绘制顺序：

```text
background
ground
decoration
pattern_path
object_points 中需要显示的静态对象
NPC / Item / Player 等动态对象
foreground
UI / HUD
```

说明：

1. 玩家角色应绘制在地面和装饰层之上；
2. 纹样空间的 `pattern_path` 可以在角色下方或上方加半透明发光效果；
3. 前景层应绘制在角色之上，用于遮挡树冠、屋檐等；
4. UI 不属于地图图层，应单独绘制；
5. `collision`、`trigger_points`、`checkpoint_points` 等对象层通常不直接绘制，只在调试模式下显示。

### 4.4 图层命名规范

推荐命名：

```text
background
ground
decoration
pattern_path
collision
object_points
npc_points
pattern_points
clue_points
checkpoint_points
trigger_points
foreground
```

不推荐命名：

```text
图层1
对象
NPC点
碰撞区域1
地图装饰
```

原因：

1. 中文命名可能导致读取不统一；
2. 图层名称不规范会影响代码解析；
3. AI/Codex 或程序读取时更容易出错；
4. 多地图协作时难以保持一致。

## 五、Tiled 对象层设计

### 5.1 对象层设计目的

对象层用于记录地图中所有可交互对象的位置信息和属性信息，包括：

1. NPC 点位；
2. 道具点位；
3. 纹样寻源点；
4. 自然线索点；
5. 纹样空间检查点；
6. 修复台；
7. 展示台；
8. 场景出口；
9. 玩家出生点；
10. 碰撞区域；
11. 隐藏收集物；
12. 任务触发区。

这样做的好处：

1. 减少硬编码；
2. 方便后期调整地图；
3. 方便 AI 和程序读取；
4. 体现对象层设计；
5. 更符合面向对象和数据驱动思想；
6. 后续新增纹样关卡时，可以优先改地图和 JSON，而不是重写主逻辑。

### 5.2 对象类型总表

| type | 所属对象层 | 说明 |
|---|---|---|
| `spawn` | `trigger_points` | 玩家出生点 |
| `scene_exit` | `trigger_points` | 场景切换出口 |
| `return_point` | `trigger_points` | 纹样空间完成后返回点 |
| `task_trigger` | `trigger_points` | 任务触发区域 |
| `npc` | `npc_points` | NPC 点位 |
| `item` | `object_points` | 可拾取物品 |
| `hidden_collectible` | `object_points` | 隐藏银纹碎片 |
| `repair_table` | `object_points` | 工坊修复台 |
| `display_table` | `object_points` | 节庆展示台 |
| `silver_part_display` | `object_points` | 银饰套装展示位 |
| `pattern_source` | `pattern_points` | 纹样寻源入口 |
| `clue` | `clue_points` | 自然场景线索点 |
| `pattern_checkpoint` | `checkpoint_points` | 纹样空间检查点 |
| `collision` | `collision` | 碰撞区域 |

### 5.3 玩家出生点对象

对象层：

```text
trigger_points
```

对象属性：

```text
id: village_spawn_start
type: spawn
spawn_id: start
```

用途：

1. 标记玩家进入当前场景时的位置；
2. 用于场景切换后放置玩家；
3. 支持多个入口对应多个出生点。

示例：

```text
village_spawn_start
village_spawn_from_workshop
village_spawn_from_market
village_spawn_from_mountain
village_spawn_from_festival
```

### 5.4 场景出口对象

对象层：

```text
trigger_points
```

对象属性：

```text
id: exit_to_workshop
type: scene_exit
target_scene: workshop
target_spawn: workshop_entry
required_task: main_02
locked_message: 先完成当前任务，再去银饰工坊。
```

用途：

1. 当玩家进入区域并按 E 时，切换到目标场景；
2. 读取 `target_scene` 和 `target_spawn`；
3. 如果有 `required_task`，则检查任务是否满足；
4. 不满足条件时显示 `locked_message`。

推荐命名：

```text
exit_to_workshop
exit_to_market
exit_to_mountain
exit_to_festival
exit_to_village
```

5. 火塘或灯笼；
6. 银饰碎片闪光点；

## 十三、摄像机与可见区域设计

### 13.1 摄像机功能

摄像机用于控制视窗在大地图中的显示范围。

它需要实现：

1. 跟随玩家；
2. 限制不越界；
3. 将世界坐标转换为屏幕坐标；
4. 只绘制当前可见区域；
5. 保证玩家始终位于视窗合理范围内。

### 13.2 摄像机计算

推荐计算方式：

```text
camera_x = player_x - viewport_width / 2
camera_y = player_y - viewport_height / 2
```

边界限制：

```text
camera_x = clamp(camera_x, 0, map_width - viewport_width)
camera_y = clamp(camera_y, 0, map_height - viewport_height)
```

### 13.3 可见瓦片范围

为了优化性能，绘制时只需要绘制当前可见瓦片。

```text
start_col = camera_x // tile_size
end_col = (camera_x + viewport_width) // tile_size + 1

start_row = camera_y // tile_size
end_row = (camera_y + viewport_height) // tile_size + 1
```

### 13.4 纹样空间摄像机规则

纹样空间可采用两种方案：

#### 方案 A：复用摄像机

如果纹样空间地图大于 800×600，继续使用 Camera 类跟随玩家。

适合：

```text
pattern_mountain.tmx
pattern_water.tmx
pattern_flower_bird.tmx
```

#### 方案 B：固定镜头

如果纹样空间地图接近视窗大小，可以固定摄像机，不跟随。

适合：

```text
pattern_sun.tmx
简单对齐型拓印界面
```

### 13.5 摄像机验收标准

1. 玩家移动时摄像机跟随；
2. 玩家靠近地图边缘时摄像机停止；
3. 屏幕不显示地图外黑边；
4. NPC 和物品绘制位置正确；
5. 交互范围不受摄像机偏移影响；
6. 纹样空间中检查点显示位置正确。

## 十四、碰撞与越界设计

### 14.1 碰撞对象

需要碰撞的对象包括：

1. 房屋；
2. 树木；
3. 摊位；
4. 墙壁；
5. 工具架；
6. 修复台；
7. 展示台；
8. 舞台边缘；
9. 地图边界；
10. 不可通行装饰物；
11. 山体边缘；
12. 河流深水区；
13. 纹样空间路径边界。

### 14.2 碰撞层制作规范

在 Tiled 中创建：

```text
collision
```

该层建议使用矩形对象绘制碰撞区域。

对象属性：

```text
type: collision
```

### 14.3 玩家碰撞规则

玩家移动时：

1. 先计算预测位置 `next_rect`；
2. 判断是否超出地图边界；
3. 判断是否与 collision 层矩形重叠；
4. 如果重叠，则取消移动；
5. 如果不重叠，则更新玩家位置。

### 14.4 越界规则

玩家世界坐标必须满足：

```text
0 ≤ player_x ≤ map_width - player_width
0 ≤ player_y ≤ map_height - player_height
```

摄像机坐标必须满足：

```text
0 ≤ camera_x ≤ map_width - viewport_width
0 ≤ camera_y ≤ map_height - viewport_height
```

### 14.5 纹样空间碰撞规则

纹样空间的碰撞不一定是“真实墙体”，可以用来约束玩家沿纹样路径移动。

设计建议：

1. 可行走区域绘制成纹样形状；
2. 路径边缘设置 collision；
3. 检查点放在路径转折处；
4. 路径宽度至少 2 个瓦片，避免操作太难；
5. 玩家偏离路径时不必 Game Over，只需被碰撞挡住或提示“沿着银色线条继续走”。

### 14.6 碰撞验收标准

1. 玩家不能穿墙；
2. 玩家不能穿过房屋；
3. 玩家不能穿过摊位；
4. 玩家不能走出地图；
5. 玩家不会卡入障碍物；
6. 关键道路不会被碰撞层误挡；
7. 纹样空间路径可走通；
8. 检查点不会放在不可达区域。

## 十八、地图制作流程

### 18.1 制作前准备

1. 确定视窗大小为 800×600；
2. 确定瓦片大小为 32×32；
3. 确定每张地图瓦片数量；
4. 准备 tileset 素材；
5. 确定地图图层结构；
6. 准备对象命名表；
7. 准备 `patterns.json`、`tasks.json`、`dialogues.json`；
8. 确定每个纹样关卡的 challenge_type。

### 18.2 Tiled 制作步骤

第一步：新建地图。

参数示例：

```text
Tile size: 32 × 32
Map size: 120 × 80
Orientation: Orthogonal
```

第二步：创建基础图层。

主地图：

```text
background
ground
decoration
collision
object_points
npc_points
pattern_points
clue_points
trigger_points
foreground
```

纹样空间：

```text
background
ground
pattern_path
collision
checkpoint_points
trigger_points
foreground
```

第三步：绘制地面层。

包括道路、草地、室内地板、山路、纹样空间路径等。

第四步：绘制装饰层。

包括房屋、树木、摊位、工具架、展示台、纹样线条等。

第五步：绘制碰撞层。

用矩形对象标记不可通行区域。

第六步：放置对象层。

放置 NPC、物品、纹样寻源点、自然线索点、检查点、出口、出生点等。

第七步：配置对象属性。

确保每个对象都有 `id` 和 `type`。

第八步：保存 TMX 文件。

保存到 `maps/` 目录。

第九步：运行游戏测试地图。

检查显示、碰撞、对象生成、纹样入口和场景切换是否正常。

### 18.3 地图检查清单

每张地图完成后检查：

1. 地图尺寸是否正确；
2. 瓦片大小是否为 32×32；
3. 图层名称是否规范；
4. NPC 是否有 id 和 type；
5. 道具是否有 id 和 type；
6. 出口是否有 target_scene 和 target_spawn；
7. 纹样寻源点是否有 pattern_id 和 challenge_type；
8. 自然线索点是否有 clue_group 和 required_count；
9. 检查点是否有 pattern_id 和 order；
10. 碰撞层是否覆盖正确；
11. 玩家出生点是否存在；
12. 关键路径是否畅通；
13. 地图是否能被程序正常加载。

5. NPC 和物品是否绘制到正确屏幕位置；

2. 玩家不能穿过树木；
3. 玩家不能穿过摊位；
4. 玩家不能穿过修复台；
5. 玩家不能走出地图；
6. 纹样空间路径边界有效；
7. 关键路线不会被碰撞层误封。

### 19.6 纹样寻源测试

测试内容：

1. 玩家靠近 `pattern_source` 时是否显示提示；
2. 按 E 是否进入对应 `challenge_type`；
3. `source_hunt` 线索是否能计数；
4. `path_maze` 检查点是否能按顺序点亮；
5. 完成后是否获得对应拓片；
6. 完成后是否返回原场景；
7. 已完成纹样是否不会重复发放奖励。

### 19.7 场景流程测试

测试流程：

```text
村寨入口
→ 与长者对话
→ 拾取银饰碎片
→ 前往工坊

→ 前往山地自然场景
```

## 二十、地图验收标准

### 20.1 基础验收标准

地图设计至少满足：

1. 有 5 个主要地图场景，或展示版本至少完成 4 个主要场景；
2. 至少有 1 个纹样空间小关卡；
3. 每张地图都有明确功能；
4. 主地图尺寸大于 800×600 视窗；
5. 瓦片大小统一为 32×32；
6. 地图像素尺寸是 32 的整数倍；
7. 图层命名规范；
8. 有碰撞层；
9. 有对象层；
10. 有玩家出生点；
11. 有场景出口。

### 20.2 功能验收标准

地图功能应满足：

1. 玩家能在地图中移动；
2. 摄像机能跟随玩家；
3. 玩家不能穿过障碍物；
4. NPC 能按对象层生成；
5. 道具能按对象层生成；
6. 纹样寻源点能按对象层生成；
7. 自然线索点能按对象层生成；
8. 纹样检查点能按对象层生成；
9. 场景出口能正确切换；
10. 纹样空间能进入和返回；
11. 关键任务路径畅通；
12. 修复台能触发工坊修复；
13. 展示台能触发结局评价。

### 20.3 设计验收标准

地图设计应满足：

1. 场景风格符合彝族银饰文化主题；
2. 地图元素服务剧情和任务；
3. 每个主场景至少有一个文化互动点；
4. 主地图路线清晰，不做复杂迷宫；
5. 纹样空间小关卡可以短而有路径挑战；
6. 地图视觉有层次；
7. 重要对象有明确视觉提示；
8. 地图不只是背景图，而是可交互游戏空间；
9. 纹样来源表达使用“游戏化转译”，不写成绝对民俗结论。

### 20.4 技术验收标准

地图技术应满足：

1. Tiled 文件能被 PyTMX 正常读取；
2. 图层可按顺序绘制；
3. 对象层属性完整；
4. 碰撞层可转换为 Rect；
5. 世界坐标和屏幕坐标转换正确；
6. 摄像机边界计算正确；
7. 地图文件路径规范；
8. 素材引用路径不混乱；
9. 纹样点、线索点、检查点读取准确；
10. 对象属性缺失时有调试提示。

## 二十二、最终说明

本结构化地图设计文档用于规范《银纹秘境：彝饰守护者》新版地图制作与程序实现。

旧版地图设计主要承载 NPC、道具、普通纹样拓印点、修复台和展示台。新版地图设计进一步承载“纹样寻源系统”，需要支持自然线索点、纹样寻源入口、纹样空间检查点、银饰套装展示位和节庆展示评价。

本项目的地图不是单纯背景图，而是承载任务、NPC、道具、纹样寻源、自然线索、纹样路径、拓片获得、修复台、展示台、碰撞区域和场景切换的核心系统。地图设计必须同时满足游戏体验和课程技术要求。

最终地图系统应体现以下能力：

1. 地图大于视窗；
2. 角色在完整地图上移动；
3. 摄像机显示地图子区域；
4. 地图采用瓦片结构；
5. 图层职责清晰；
6. 对象层驱动 NPC、道具和触发点；
7. 对象层驱动纹样寻源点、自然线索点和检查点；
8. 碰撞和越界处理正确；
9. 场景连接服务主线流程；
10. 纹样空间服务“走出纹样”的小关卡；
11. 地图文化元素服务彝族银饰主题；
12. 整体结构便于后续编码、测试和验收。
