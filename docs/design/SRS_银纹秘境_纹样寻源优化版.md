# 《银纹秘境：彝饰守护者》软件需求规格说明书 SRS（纹样寻源优化版）

## 文档说明

### 1.1 编写目的

本文档用于描述《银纹秘境：彝饰守护者》游戏系统的软件需求，包括系统目标、功能需求、非功能需求、外部接口需求、数据需求、运行环境、技术约束、测试与验收标准等内容。

本版本根据新版游戏策划进行同步调整，重点将原有“纹样拓印”升级为“纹样寻源系统”，将原有“修复一件银饰”升级为“修复一套节庆银饰”，并加入纹样小关卡、敲银修复、银光值、套装进度和结局评价等内容。

本文档面向以下对象：

1. 项目开发者；
2. 课程指导教师；
3. AI/Codex 协作开发工具；
4. 后续维护人员；
5. 项目验收与测试人员。

### 1.2 项目背景

《银纹秘境：彝饰守护者》是一款以彝族银饰文化为主题的 2D 单人文化修复探索 RPG 游戏。玩家扮演“银纹守护者”，在村寨、山地、集市、银饰工坊和节庆广场中探索，寻找银饰碎片，理解纹样来源，进入纹样空间完成小关卡，获得纹样拓片，并最终使用拓片和修复材料完成节庆银饰套装的修复。

本项目不是简单的静态文化展示页面，也不是传统打怪升级 RPG，而是一个具有地图系统、视窗系统、角色移动、NPC 对话、对象层读取、任务推进、道具收集、纹样寻源、敲银修复、图鉴解锁、银光值评价和节庆展示的 2D 游戏 Demo。

### 1.3 项目范围

本系统需要完成一个可运行、可展示、可验收的课程级 2D 游戏项目。

系统范围包括：

1. 游戏启动与主菜单；

### 1.4 术语说明

| 术语 | 说明 |
|---|---|
| 纹样寻源 | 玩家通过自然场景、生活场景或纹样空间小关卡理解纹样来源并获得拓片的过程。 |
| 纹样拓片 | 玩家完成纹样寻源后获得的道具，用于后续银饰修复。 |
| 纹样空间 | 一个小型关卡场景，玩家通过走纹样路径、对齐拓印纸或描摹关键点完成纹样获得。 |
| 走纹样迷宫 | 玩家操作角色沿纹样形状的路径移动，按顺序经过检查点后生成拓片。 |
| 自然场景寻源 | 玩家在自然或生活场景中收集线索，理解纹样来源后获得拓片。 |
| 敲银修复 | 玩家在工坊中使用拓片和材料，通过时机判定完成银饰加工。 |
| 银光值 | 评价玩家修复表现、图鉴进度、隐藏收集和支线完成度的分数。 |
| 节庆银饰套装 | 玩家需要逐步修复的银饰部件集合，如银耳环、银手镯、银项圈、银铃或银链。 |
| 游戏化转译 | 将自然意象、生活场景和银饰纹样关系转化为游戏机制，不将缺少明确出处的寓意写成绝对民俗结论。 |

## 2. 总体描述

### 2.1 产品定位

本系统是一款基于 Python + Pygame 的 2D 单人离线文化修复探索游戏。

产品定位为：

1. 课程项目；
2. 文化主题游戏 Demo；
3. 非战斗型探索与修复游戏；
4. 轻量剧情任务游戏；
5. AI 协作开发实践项目。

### 2.2 产品目标

本系统应实现以下目标：

1. 提供完整的游戏开始、游玩、结局流程；
2. 支持玩家在大于视窗的 2D 地图中移动；
3. 支持角色、NPC、道具、任务、图鉴等对象的面向对象封装；
4. 支持地图图层、对象层、碰撞层、线索点、纹样点和检查点读取；
5. 支持与彝族银饰文化相关的任务、纹样寻源、拓片收集、敲银修复和图鉴玩法；
6. 支持节庆银饰套装进度、银光值和结局评价；
7. 提供清晰的用户交互和操作反馈；
8. 满足课程项目对专业性、规范性和完成度的要求。

### 2.3 用户特征

目标用户包括：

1. 小学生和初中生；
2. 对传统文化感兴趣的学生；
3. 课程展示中的老师和同学；
4. 第一次接触彝族银饰文化的普通玩家；
5. 喜欢轻量探索、收集和解谜玩法的玩家。

用户特点：

1. 不希望操作复杂；
2. 希望任务目标明确；
3. 更容易接受图像化、互动式内容；
4. 对收集、图鉴、成就反馈敏感；
5. 希望能在较短时间内完成一次完整体验；
6. 可以接受轻量判断、路径行走和时机判定，但不适合高强度战斗。

### 2.4 运行环境

#### 2.4.1 硬件环境

最低硬件要求：

1. 普通 Windows PC；
2. 支持 Python 运行环境；
3. 显示器分辨率不低于 800 × 600；
4. 键盘和鼠标；
5. 基础音频输出设备。

#### 2.4.2 软件环境

推荐软件环境：

1. 操作系统：Windows 10 或 Windows 11；
2. 编程语言：Python 3.x；
3. 游戏框架：Pygame；
4. 地图编辑器：Tiled Map Editor；
5. 地图读取库：PyTMX；
6. 数据文件格式：JSON；
7. 开发工具：VS Code 或 PyCharm；
8. 版本管理工具：Git，可选。

### 2.5 设计与实现约束

系统开发应满足以下约束：

1. 必须使用 Python + Pygame 作为主要开发技术；
2. 游戏应为 2D 离线单人游戏；
3. 地图设计应优先采用 32 × 32 像素瓦片；
4. 推荐视窗大小为 800 × 600 像素；
5. 地图尺寸应大于视窗，体现滚动探索效果；
6. 地图尺寸应尽量为瓦片大小的整数倍；
7. 代码应采用面向对象设计；
8. 不得将所有功能集中写在 main.py 中；
9. 应区分地图层、对象层、角色层、前景层和 UI 层；
10. 应实现基本碰撞检测和越界处理；
11. 应避免引入复杂第三方依赖；
12. 应优先保证主流程完整，再扩展支线、美术、音效和高级拓印玩法；
13. 纹样寻源优先实现 source_hunt 与 path_maze，align_rubbing 和 point_trace 可作为优秀扩展；
14. 文化内容应保持简短、尊重、准确，对缺少明确出处的寓意使用“游戏化转译”表达。

### 2.6 假设与依赖

本系统开发基于以下假设：

1. 开发者能够安装并运行 Python 与 Pygame；
2. 地图素材可通过 Tiled 进行编辑；
3. 缺少正式美术素材时，可使用占位图形或简单色块代替；
4. 游戏不依赖网络服务；
5. 文化说明文本以简短、尊重、准确为原则；
6. 项目最终以课程展示和本地运行验收为主；
7. 纹样寻源的路径、线索点和检查点可通过对象层或 JSON 配置。

## 3. 外部接口需求

#### UI-01 主菜单界面

系统应提供主菜单界面。

主菜单应包含：

4. 操作说明按钮，可选；
5. 退出游戏按钮。

#### UI-02 主游戏界面

主游戏界面应显示：

1. 当前地图；
2. 玩家角色；
3. NPC；
4. 可拾取物品；
5. 纹样寻源点；
6. 当前任务提示；
7. 交互提示；
8. 节庆银饰套装进度；
9. 银光值；
10. 图鉴进度。

#### UI-03 对话界面

系统应提供 NPC 对话框。

对话框应显示：

1. NPC 名称；
2. 对话文本；
3. 继续提示；
4. 任务状态变化提示，可选；
5. 纹样来源线索，可选；
6. 缺少材料提示，可选。

#### UI-04 背包界面

系统应提供背包界面。

背包界面应显示：

1. 已获得物品列表；
2. 物品名称；
3. 物品说明；
4. 纹样拓片类型；
5. 银饰材料数量或拥有状态；
6. 返回按钮或关闭提示。

系统应提供银饰与纹样图鉴界面。

图鉴界面应显示：

1. 已解锁银饰条目；
2. 已解锁纹样条目；

系统应提供纹样寻源相关界面。

不同 challenge_type 对应界面内容：

系统应提供银饰工坊修复界面。

4. 所需纹样拓片；

6. 敲银命中次数；
7. 银饰亮度变化；
8. 成功反馈；
9. 错误提示。

#### UI-08 结束与结局评价界面

系统应提供结束界面。

结束界面应显示：

1. 玩家称号；
2. 银饰套装完成度；
3. 图鉴收集数量；
4. 银光值；
5. 支线任务完成情况；
6. 结局评价；
7. 重新开始按钮；
8. 退出按钮。

### 3.2 输入接口需求

#### IN-01 键盘输入

系统应支持以下键盘操作：

1. W / A / S / D 或方向键：控制玩家移动；
2. E：交互、对话、拾取、进入纹样寻源点；
3. I：打开或关闭背包；
4. G：打开或关闭图鉴；
5. Esc：返回、关闭界面或暂停；
6. 空格：敲银时机判定确认，可选；
7. Enter：确认菜单或对话推进，可选。

#### IN-02 鼠标输入

系统应支持以下鼠标操作：

1. 点击开始游戏；
2. 点击退出游戏；
3. 点击图鉴条目；
4. 点击修复部件或材料选项；
5. 在纹样对齐玩法中拖动或调整拓印纸，可选；
6. 在关键点描摹玩法中点击或拖过关键点，可选；
7. 点击重新开始。

### 3.3 软件接口需求

系统应与以下软件或文件接口配合：

1. Pygame：负责窗口、绘制、事件、音效和主循环；
2. Tiled：负责编辑地图、图层、对象层、线索点和检查点；
3. PyTMX：负责读取 TMX 地图文件；
4. JSON 文件：负责存储 NPC、任务、物品、图鉴、对话、纹样配置、结局配置等数据；
5. 本地文件系统：负责加载图片、音频、地图和数据文件。

### 3.4 通信接口需求

本系统为单人离线游戏，不需要网络通信接口。

## 4.2 游戏启动与菜单模块

### FR-MENU-01 启动游戏

系统应在运行 main.py 后初始化 Pygame，并打开游戏窗口。

验收标准：

1. 程序可正常启动；
2. 窗口大小符合配置；
3. 不出现启动崩溃；
4. 能进入主菜单界面。

### FR-MENU-02 显示主菜单

系统应显示主菜单界面，包括游戏标题、开始按钮和退出按钮。

验收标准：

1. 标题显示正确；
2. 开始按钮可用；
3. 退出按钮可用；
4. 界面不会遮挡或错位。

### FR-MENU-03 开始游戏

当玩家点击开始按钮或按确认键时，系统应进入剧情引导或第一个游戏场景。

验收标准：

1. 玩家可以从主菜单进入游戏；
2. 游戏状态从菜单状态切换为游玩状态；
3. 不出现黑屏或卡死。

### FR-MENU-04 退出游戏

当玩家点击退出按钮或关闭窗口时，系统应正常退出。

验收标准：

1. 程序能正常结束；
2. 不产生未处理异常。

## 4.3 地图与场景模块

### FR-MAP-01 加载地图

系统应支持加载 Tiled 制作的 TMX 地图文件。

验收标准：

1. 能加载 village、workshop、market、mountain、festival 等主要地图；
2. 能加载 pattern_mountain、pattern_water、pattern_sun 等纹样空间地图，可选；
3. 地图显示正确；
4. 图层顺序正确；
5. 地图加载失败时应有错误提示。

### FR-MAP-02 支持地图大于视窗

系统应支持地图尺寸大于游戏视窗。

推荐参数：

1. 视窗：800 × 600 像素；
2. 瓦片：32 × 32 像素；
3. 村寨地图：3840 × 2560 像素；
4. 工坊地图：1280 × 960 像素；
5. 集市地图：2560 × 1920 像素；
6. 山地地图：2560 × 1920 像素；
7. 广场地图：1920 × 1280 像素；
8. 纹样空间地图：可小于主地图，但应支持碰撞与检查点。

验收标准：

1. 玩家只能看到地图的一部分；
2. 玩家移动时视窗能跟随变化；
3. 不显示地图外空白区域。

### FR-MAP-03 支持多图层绘制

系统应支持以下图层：

1. background；
2. ground；
3. decoration；
4. collision；
5. object_points；
6. npc_points；
7. pattern_points；
8. clue_points；
9. checkpoint_points；
10. trigger_points；
11. foreground。

验收标准：

1. 地面层在角色下方；
2. NPC 和玩家显示在角色层；
3. 前景层可遮挡角色；
4. UI 不写入地图层；
5. 纹样寻源点、线索点和检查点能从对象层读取。

### FR-MAP-04 支持场景切换

系统应支持玩家通过触发区在场景之间切换。

验收标准：

1. 玩家进入出口区域后切换场景；
2. 切换后玩家出现在目标出生点；
3. 场景切换后任务、背包、图鉴、银光值状态不丢失；
4. 不出现坐标错误或黑屏。

### FR-MAP-05 支持纹样空间场景

系统应支持从主地图进入纹样空间小关卡。

验收标准：

1. 玩家与纹样寻源点交互后可进入对应纹样空间；
2. 系统记录返回场景和返回坐标；
3. 完成或退出纹样空间后可返回原场景；
4. 纹样空间中的碰撞、检查点和 UI 能正常工作。

## 4.4 摄像机与视窗模块

### FR-CAM-01 摄像机跟随玩家

系统应实现摄像机跟随玩家移动。

验收标准：

1. 玩家在地图中部移动时，视窗随玩家移动；
2. 玩家不应长期离开屏幕中心区域；
3. 摄像机移动平滑或基本稳定。

### FR-CAM-02 摄像机边界限制

系统应限制摄像机不能显示地图外区域。

计算规则：

```text
0 ≤ camera_x ≤ map_width - viewport_width
0 ≤ camera_y ≤ map_height - viewport_height
```

验收标准：

1. 摄像机不会越过地图左边界；
2. 摄像机不会越过地图右边界；
3. 摄像机不会越过地图上边界；
4. 摄像机不会越过地图下边界；
5. 视窗不显示黑边或空白地图外区域。

### FR-CAM-03 坐标转换

系统应支持世界坐标到屏幕坐标的转换。

计算规则：

```text
screen_x = world_x - camera_x
screen_y = world_y - camera_y
```

验收标准：

1. 玩家、NPC、物品、纹样点在屏幕中显示位置正确；
2. 摄像机移动后对象显示不发生错位；

## 4.6 碰撞检测模块

### FR-COLL-01 加载碰撞区域

验收标准：

1. 房屋、树木、摊位、墙体等不可穿越区域能被读取；
2. 纹样空间路径边界能被读取；

验收标准：

1. 若新位置发生碰撞，取消该方向移动；

## 4.10 纹样寻源系统模块

### FR-PATTERN-01 生成纹样寻源点

系统应从 pattern_points 或 object_points 层读取纹样寻源点。

纹样寻源点数据至少包括：

1. id；
2. name；
3. type；
4. pattern_id；
5. challenge_type；
6. reward_item；
7. target_scene，可选；
8. codex_unlock，可选；
9. required_task，可选。

验收标准：

1. 纹样寻源点能显示在指定场景；
2. 纹样寻源点具有 pattern_id 和 challenge_type；
3. 玩家靠近后可交互；
4. 对象数据可配置。

### FR-PATTERN-02 进入纹样寻源

玩家靠近纹样寻源点后，系统应显示交互提示。

提示示例：

```text
按 E 进入纹样寻源
```

验收标准：

1. 玩家靠近后显示提示；
2. 按 E 后进入对应挑战；
3. 未满足任务条件时应显示提示；
4. 进入挑战后主地图操作暂停或切换到纹样空间状态。

### FR-PATTERN-03 支持 challenge_type 配置

系统应支持至少以下 challenge_type：

1. source_hunt：自然场景寻源；
2. path_maze：走纹样迷宫；
3. align_rubbing：纹样对齐，可选；
4. point_trace：关键点描摹，可选。

验收标准：

1. 系统能根据 challenge_type 调用对应流程；
2. 不同 challenge_type 不需要修改主逻辑即可扩展；
3. 未识别类型应给出错误提示或使用默认处理。

### FR-PATTERN-04 自然场景寻源

系统应支持 source_hunt 类型纹样寻源。

流程要求：

1. 系统读取 clue_points；
2. 玩家靠近线索点后按 E 记录；
3. 系统统计同一 clue_group 的收集数量；
4. 达到 required_count 后获得对应纹样拓片；
5. 任务、背包、图鉴状态更新。

验收标准：

1. 线索点能从对象层读取；
2. 玩家能记录线索；
3. 已记录线索不会重复计数；
4. 线索收集完成后获得拓片；
5. 获得拓片后能用于修复条件判断。

### FR-PATTERN-05 走纹样迷宫

系统应支持 path_maze 类型纹样寻源。

流程要求：

1. 系统加载纹样空间地图；
2. 地图中存在 checkpoint_points；
3. 玩家按顺序经过检查点；
4. 每经过一个检查点，对应线条或节点发光；
5. 全部检查点完成后生成纹样拓片；
6. 系统返回原场景或提示玩家返回。

验收标准：

1. 纹样空间地图能加载；
2. 检查点顺序能正确判断；
3. 玩家走错或漏掉检查点时有提示；
4. 完成后获得对应拓片；
5. 重复完成不会重复发放奖励。

### FR-PATTERN-06 纹样对齐，可选

系统可支持 align_rubbing 类型纹样寻源。

流程要求：

1. 系统显示底层浅色纹样；
2. 系统显示可移动半透明拓印纸；
3. 玩家使用方向键或鼠标微调位置；
4. 系统根据偏移距离或模拟重合度判断结果；
5. 达到标准后获得普通或清晰拓片。

验收标准：

1. 纹样与拓印纸能显示；
2. 玩家能调整拓印纸位置；
3. 重合度达到阈值后成功；
4. 清晰拓片可提供银光值加成，可选。

### FR-PATTERN-07 关键点描摹，可选

系统可支持 point_trace 类型纹样寻源。

流程要求：

1. 系统显示若干纹样关键点；
2. 玩家用鼠标按顺序点击或拖过关键点；
3. 系统判断顺序和距离；
4. 完成后生成纹样线条和拓片。

验收标准：

1. 关键点显示清楚；
2. 玩家能按顺序完成；
3. 错误顺序有提示；
4. 完成后获得拓片。

### FR-PATTERN-08 拓片奖励与状态更新

完成纹样寻源后，系统应发放拓片并更新状态。

验收标准：

1. 背包添加 reward_item；
2. 图鉴解锁对应纹样条目；
3. 当前任务推进；
4. 纹样寻源点状态改为 completed；
5. 已完成纹样不会重复发放奖励；
6. 系统显示“获得 xxx 拓片”反馈。

## 4.11 银饰工坊修复模块

### FR-REPAIR-01 进入工坊修复

前置条件包括：

验收标准：

1. 未满足条件时给出提示；
2. 满足条件后可进入修复界面；
3. 进入修复界面后暂停主地图操作。

### FR-REPAIR-02 选择银饰部件

系统应允许玩家选择要修复的银饰部件。

银饰部件建议包括：

1. 银耳环；
2. 银手镯；
3. 银项圈；
4. 银铃或银链。

验收标准：

1. 界面显示可修复部件；
2. 未解锁部件显示为不可修复或隐藏；
3. 已修复部件显示为已点亮；
4. 选择部件后显示所需材料和拓片。

### FR-REPAIR-03 材料与拓片检查

系统应检查玩家是否拥有对应材料和纹样拓片。

验收标准：

1. 材料齐全时允许进入敲银；
2. 材料不足时显示缺少内容；
3. 拓片不足时提示前往对应纹样寻源；
4. 检查结果不应误判。

### FR-REPAIR-04 敲银时机判定

系统应实现敲银时机判定小游戏。

流程要求：

1. 屏幕显示敲银进度条或指针；
2. 银色判定区周期性出现；
3. 玩家在指针进入判定区时按空格或点击确认；
4. 命中后增加修复进度；
5. 达到指定命中次数后修复成功；
6. 未命中时给予提示并允许继续尝试。

验收标准：

1. 敲银界面能显示；
2. 玩家输入能被正确识别；
3. 命中和未命中有不同反馈；
4. 成功后银饰部件点亮；
5. 失败不导致任务卡死；
6. 失败可影响银光值或修复评分。

### FR-REPAIR-05 银饰状态变化

修复成功后，系统应使银饰状态发生变化。

银饰状态包括：

1. 暗淡；
2. 微光；
3. 发亮；
4. 闪耀。

验收标准：

1. 修复成功后银饰至少从暗淡变为发亮；
2. 修复完整套装或高评价时可变为闪耀；
3. 反馈应包括视觉变化；
4. 可加入金属敲击音效。

### FR-REPAIR-06 套装进度更新

系统应记录节庆银饰套装进度。

验收标准：

1. 每修复一个新部件，套装进度增加；
2. 已修复部件不会重复计数；
3. HUD 显示套装进度；
4. 套装进度可用于结局评价。

### FR-SCORE-02 银光值变化

系统应根据玩家表现调整银光值。

银光值变化依据包括：

1. 修复银饰部件；
2. 获得清晰拓片；
3. 完成支线任务；
4. 找到隐藏银纹；
5. 敲银失败次数；
6. 纹样小关卡表现。

验收标准：

1. 完成正向行为可提高银光值；
2. 错误行为可轻微降低银光值；
3. 关键道具不因失败被扣除；
4. 银光值变化有提示或能在结局中体现。

### FR-END-01 进入节庆展示

玩家完成至少一个银饰部件修复后，应能前往节庆广场触发展示。标准版本可要求至少 2 个部件，优秀版本可要求 3 到 4 个部件。

验收标准：

1. 未完成最低修复条件时不能触发最终结局；
2. 完成条件后可进入展示流程；
3. 与节庆主持人对话后进入结局界面。

### FR-END-02 结局评价

系统应根据玩家完成情况给出评价。

评价依据包括：

1. 银饰套装完成度；
2. 银光值；
3. 图鉴解锁数量；
4. 隐藏银纹碎片数量；
5. 支线任务完成情况；
6. 敲银修复表现；
7. 拓片质量，可选。

结局类型建议：

1. 普通结局；
2. 良好结局；
3. 完整守护者结局。

验收标准：

1. 至少实现一个基础结局；
2. 标准版本应支持根据完成度显示不同评价；
3. 结局界面显示玩家称号、银光值和完成度；
4. 结局评价与玩家行为存在对应关系。

### FR-END-03 重新开始或退出

结局界面应提供重新开始和退出功能。

验收标准：

1. 点击重新开始可返回主菜单或重新进入游戏；
2. 点击退出可关闭程序。

## 6. 非功能需求

### 6.1 性能需求

#### NFR-PERF-01 帧率

系统应在普通 PC 上保持基本流畅运行。

验收标准：

1. 目标帧率为 30 FPS 或以上；
2. 玩家移动时不应明显卡顿；
3. 地图绘制应优先绘制视窗内可见瓦片，避免每帧绘制整张大地图；
4. 纹样空间小地图不应明显降低帧率。

#### NFR-PERF-02 加载性能

系统加载地图和资源时应尽量避免长时间无响应。

验收标准：

1. 单个场景加载时间应在可接受范围内；
2. 资源加载失败时有错误提示；
3. 不应因缺少单个非关键资源导致整个游戏崩溃。

### 6.2 易用性需求

#### NFR-USE-01 操作简单

系统应保证玩家能快速理解基础操作。

验收标准：

1. 游戏内有操作提示；
2. 交互对象附近有提示；
3. 主要操作不超过常用键位范围；
4. 纹样小关卡规则简单直观；
5. 敲银小游戏规则简单直观。

#### NFR-USE-02 任务清晰

系统应持续显示当前任务目标。

验收标准：

1. 玩家能知道下一步要做什么；
2. 任务完成后提示应更新；
3. 未满足条件时应给出明确提示；
4. 线索收集、检查点进度和套装进度应可理解。

### 6.3 可靠性需求

#### NFR-REL-01 防止卡死

系统应避免任务流程卡死。

验收标准：

1. 关键任务有清晰触发条件；
2. 任务未完成时有提示；
3. 纹样小关卡允许重新尝试；
4. 敲银失败允许重新尝试；
5. 关键道具不因失败永久丢失；
6. 玩家不会因遗漏支线任务无法通关。

#### NFR-REL-02 异常处理

系统应处理常见异常。

包括：

1. 地图文件缺失；
2. 图片文件缺失；
3. 音效文件缺失；
4. JSON 数据格式错误；
5. 无效对象层属性；
6. 缺少纹样配置；
7. 检查点顺序错误。

验收标准：

1. 关键错误有提示；
2. 非关键资源缺失时可用占位图替代；
3. 程序尽量不直接崩溃；
4. 调试模式能辅助定位对象层配置问题。

### 6.4 可维护性需求

#### NFR-MAINT-01 模块化结构

系统应采用模块化文件结构。

验收标准：

1. main.py 只负责启动；
2. 玩家、地图、NPC、任务、背包、图鉴、纹样寻源、修复小游戏、评分、UI 等模块分离；
3. 类职责清晰；
4. 关键代码有注释。

#### NFR-MAINT-02 数据与逻辑分离

系统应尽量将对话、任务、物品、图鉴、纹样、银饰部件和结局数据放入 JSON 文件。

验收标准：

1. 修改 NPC 对话不需要大量修改代码；
2. 修改任务文本不需要重写主逻辑；
3. 新增物品、纹样或图鉴条目较方便；
4. 新增纹样寻源关卡时优先修改地图对象和 JSON 配置。

### 6.5 可移植性需求

系统主要面向 Windows PC 本地运行。

验收标准：

1. 在 Windows 环境可运行；
2. 项目目录结构清楚；
3. 依赖库可通过 requirements.txt 记录；
4. 后续可使用 PyInstaller 打包为 exe，可选。

### 6.6 安全性需求

本系统不涉及账号、密码、支付、联网和个人隐私数据。

安全需求包括：

1. 不读取用户敏感文件；
2. 不上传任何数据；
3. 不进行网络通信；
4. 存档仅保存在本地项目目录或指定目录。

### 6.7 文化表达规范需求

系统中的文化内容应保持尊重和准确。

验收标准：

1. 不编造复杂民俗细节；
2. 不将彝族银饰文化猎奇化；
3. 不使用不尊重的表达；
4. 文化说明应简短、准确、正向；
5. 文化内容应与游戏行为绑定，而不是单独堆砌文本；
6. 对缺少明确出处的纹样寓意，使用“游戏化转译”表述；
7. 文化资料用于支撑方向，游戏机制负责互动转译。

6. 失败后可以重新尝试；
7. 文化内容与玩法相关；
8. 结尾有完成感；
9. 银光值或结局评价能体现玩家表现。

### 9.4 文档验收

项目文档应包含：

1. 游戏策划文档；
2. 软件需求规格说明书；
3. 功能规格说明书；
4. 技术设计说明；
5. 测试计划；
6. 资产清单；
7. AI 协作开发说明，可选；
8. 运行说明。

### 9.5 展示验收重点

课程展示时建议重点展示：

1. 玩家在大地图中移动与碰撞；
2. 与 NPC 对话获得纹样线索；
3. 进入纹样寻源小关卡；
4. 通过走纹样迷宫或自然场景寻源获得拓片；
5. 拓片进入背包；
6. 返回工坊使用拓片和材料进行敲银修复；
7. 银饰部件点亮；
8. 图鉴解锁；
9. 节庆广场根据完成度给出评价。

## 10. 测试需求

### 10.1 地图测试

测试项：

1. 地图是否能加载；
2. 瓦片是否对齐；
3. 图层顺序是否正确；
4. 地图是否大于视窗；
5. 视窗是否只显示地图子区域；
6. 纹样空间地图是否能加载。

### 10.2 摄像机测试

测试项：

1. 摄像机是否跟随玩家；
2. 摄像机是否不会越界；
3. 玩家靠近地图边缘时是否正常；
4. 世界坐标到屏幕坐标是否正确。

### 10.3 碰撞测试

测试项：

1. 玩家是否不能穿墙；
2. 玩家是否不能穿过房屋；
3. 玩家是否不能穿过树木和摊位；
4. 玩家是否不能走出地图；
5. 纹样空间路径边界是否有效；
6. 碰撞层是否读取正确。

### 10.4 NPC 测试

测试项：

1. NPC 是否按对象层点位生成；
2. 玩家靠近 NPC 是否出现提示；
3. 按 E 是否能触发对话；
4. NPC 对话是否随任务状态变化；
5. NPC 是否能提供纹样来源线索。

### 10.5 物品测试

测试项：

1. 物品是否按对象层点位生成；
2. 玩家靠近物品是否出现提示；
3. 按 E 是否能拾取；
4. 拾取后物品是否进入背包；
5. 拾取后物品是否从地图消失；
6. 拾取是否能触发任务或图鉴变化。

### 10.6 任务测试

测试项：

1. 主线任务是否按顺序推进；
2. 未完成前置任务时是否有提示；
3. 完成任务后提示是否更新；
4. 是否能从第一任务推进到最终结局；
5. 银饰套装进度是否正确更新。

### 10.7 纹样寻源测试

测试项：

1. 纹样寻源点是否能生成；
2. 玩家靠近是否显示交互提示；
3. 按 E 是否进入对应小关卡；
4. source_hunt 是否能统计线索数量；
5. path_maze 是否能按顺序点亮检查点；
6. align_rubbing 是否能判断重合度，可选；
7. point_trace 是否能判断关键点顺序，可选；
8. 完成后是否获得对应拓片；
9. 拓片是否进入背包；
10. 重复完成是否不会重复发放奖励；
11. 拓片是否能用于修复小游戏。

### 10.8 修复小游戏测试

测试项：

1. 是否能进入工坊修复界面；
2. 是否能选择银饰部件；
3. 材料不足是否有提示；
4. 拥有拓片和材料时是否能开始敲银；
5. 敲银命中是否增加进度；
6. 敲银失败是否允许重试；
7. 成功后是否更新任务状态；
8. 成功后是否提升套装进度和银光值；
9. 成功后是否解锁图鉴。

### 10.9 图鉴测试

测试项：

1. 图鉴是否能打开；
2. 未解锁条目是否隐藏；
3. 完成纹样寻源后是否解锁纹样图鉴；
4. 完成银饰修复后是否解锁银饰图鉴；
5. 图鉴内容是否正确显示；
6. 图鉴进度是否正确统计。

### 10.10 结局评价测试

测试项：

1. 是否能进入节庆广场；
2. 不同银饰修复数量是否产生不同展示结果；
3. 银光值是否正确计算；
4. 图鉴进度是否影响结局；
5. 支线和隐藏银纹是否影响评价；
6. 重新开始和退出是否可用。

### 10.11 完整流程测试

测试流程：

1. 启动游戏；
2. 进入主菜单；
3. 开始游戏；
4. 与村寨长者对话；
5. 拾取第一块银饰碎片；
6. 前往银饰工坊；
7. 与银匠师傅对话；
8. 前往山地或集市；
9. 完成至少一个纹样寻源小关卡；
10. 获得纹样拓片；
11. 收集修复材料；
12. 返回工坊；
13. 完成敲银修复；
14. 点亮银饰部件；
15. 解锁图鉴；
16. 前往节庆广场；
17. 触发结局评价；
18. 进入结局界面。

## 12. 最终说明

本 SRS 文档将《银纹秘境：彝饰守护者》的新版游戏策划内容转化为软件开发需求。文档重点不再只是描述游戏创意，而是明确系统需要实现的功能、接口、数据、性能、可维护性、测试方法和验收标准。

本项目最终应完成一个主题明确、结构规范、功能闭环、能够运行展示的 2D Pygame 非战斗型文化修复探索游戏。系统应围绕彝族银饰文化展开，以地图探索、纹样寻源、纹样空间小关卡、拓片获得、敲银修复、图鉴解锁、银光值评价和节庆展示为核心功能，同时满足课程对地图、视窗、图层、对象层、精灵类、碰撞检测、越界处理和面向对象设计的要求。

新版需求的核心变化是：

1. 将“纹样拓印”升级为“纹样寻源系统”；
2. 将“按 E 获得拓片”升级为“自然场景寻源、走纹样迷宫、纹样对齐和关键点描摹”等小关卡；
3. 将“修复一件银饰”升级为“修复一套节庆银饰”；
4. 将“修复成功”升级为“套装进度、银光值和多结局评价”；
5. 将文化内容从静态说明转化为可操作、可反馈、可测试的游戏机制。
